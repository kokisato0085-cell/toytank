# 開発ルール

本リポジトリの開発で守るルール。ウォーターフォール（設計先行）で進めながら、技術的障壁とその解決策を記録として積み上げることを目的とする。

## R1. リポジトリ隔離
- 本プロジェクトのファイルは、すべてこのリポジトリ内にまとめる。
- 他のプロジェクトのファイルと混在させない。

## R2. 自己完結
- このリポジトリ単体で完結させる。
- 本プロジェクトに無関係なファイルはコミットしない。

## R3. 単体テスト
- 機能実装の単位ごとに単体テストを作成する。
- テストは実装と並走させ、こまめに積み上げる。

## R4. 進捗の明示
- README 等に、進捗状況と現在の課題・問題点を細かく記載し続ける。
- **直面した技術的障壁と、その解決策を必ずセットで記録する**（本プロジェクトの中心的な記録方針）。

## R5. 静的解析・コードスタイル・セキュリティスキャン
- 静的解析ツール（リンター／フォーマッター等）を導入する。
- コードスタイルを設定ファイルでルール化し、自動で整える。
- （具体的なツールは技術スタック確定時に選定する。例: TypeScript なら ESLint + Prettier）
- **CIでセキュリティスキャンを常時実行する**（GitHub Actions `.github/workflows/security.yml`）。
  - **semgrep**：静的解析（コードの危険箇所・脆弱性パターン検知）。標準ルール `p/typescript` `p/javascript` `p/security-audit`。
  - **gitleaks**：コミット履歴・差分からシークレット（APIキー・パスワード等）の漏れを検知。
  - push（main）と Pull Request で実行し、**問題検出時はCIを失敗させて止める**。

## R6. 可読性・自己説明的なコード
- 誰が読んでも分かりやすいコードを書く。
- **自己説明的**（命名で意図が伝わる）にし、コメントは「なぜ」を説明する。
- **関心の分離**：層・責務を分け、1つの関数/型/パッケージは1つの責務に集中する。
- **一貫性**：命名・構造・エラー処理の流儀をプロジェクト全体で揃える。

## R7. テスト駆動（設計書ベース）
- **設計書に基づくテストを先に書き**、それを満たす実装を行い、その後リファクタリングする。
- テストは仕様（設計書）の表現であり、リファクタリングの安全網とする。

## R8. 設計意図の記録
- **複雑なロジックには意図を残す**：コード上のコメントに加え、必要に応じて README 等に「なぜその手法か／どう動くか」を記述する習慣を持つ。
- 静的解析（R5）は言語標準（Go は `go vet`＋`gofmt`、TS は strict＋型チェック）を常時通す。
