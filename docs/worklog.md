### タスク1: E＝最大4人の実接続（host + guest×3）
- **背景**: 内部モデルは既に N 人対応済みだったが、実接続（relay・ロビー・入力配線）が 2 人（host + guest1）止まりだった。
- **設計判断（ユーザー承認）**:
  1. ゲスト離脱 → 残りは続行・その戦車は退場／ホスト離脱 → 全員終了。
  2. 2〜4 人でホストが任意に開始・開始後の途中参加なし。
- **設計反映**: §12-l 新設・§12-f 改定・POLICY 大3/大7 更新。
- **実装要点**:
  - relay: ゲストへスロットid(1〜3)割当（タグ `s1`〜`s3`＋`getTags`）→ `joined.id`/`peer-joined.id` で通知。満室(guest3)で `full`。ルーティング = host→全ゲストへブロードキャスト／各ゲスト→host のみ。切断 = ホスト離脱で全終了・ゲスト離脱は該当のみ `peer-left{id}`。スロット割当を純関数 `nextFreeSlot`（`relay/src/slot.ts`）に分離。
  - client: `net/relay.ts` に `myId`。`game.ts` は remote 入力を id 別 Map 化・`coopSpawns` を N 人化（P3/P4 は P1 隣に自動配置）・players 配列を `playerById`/`localPlayer` で id 引き（中抜け耐性）・ゲスト側は `ensureGuestPlayer`/`pruneGuestPlayers` で全員動的描画＋退場者剪定・`removeCoopPlayer`・色 p3/p4・HUD/リザルトを N 人化。`main.ts` は多ゲスト id 割当・roster 権威管理・2〜4 人でホスト任意開始・ゲスト退場処理。
- **テスト/検証**:
  - relay: `nextFreeSlot` の単体テスト5件（vitest を relay に導入）。
  - client: `tsc` / vitest 50件 / `vite build` OK。
  - Node 組込み WebSocket でローカル/本番 relay に host+guest×3 を実接続し、割当・満室・スロット再利用・ホスト離脱ブロードキャストを確認。
  - ユーザーがローカル 4 窓で実プレイ確認。
- **不具合修正（同セッション）**:
  - リスタート時に host と一部プレイヤーしか復帰しない → `restartCoop` が `startCoopHost(stage)` を引数なしで呼び `coopIds` が既定 `[0,1]` に潰れていた。`this.coopIds` を明示的に渡して修正。
  - id3（4人目）が壁に埋まる面 → 自動配置オフセットを下(+y)→上(-y)へ変更。
- **対応**: Issue #83 → PR #84（main=`7b2e597`）。**本番は relay 再デプロイ → merge の順**で反映（旧 relay は `joined.id` を返さずゲスト id が host と衝突するため、client/relay はセット反映が必須）。

### 3b 体感調整（補間遅延）
- **方針（ユーザー承認）**: 計測オーバーレイは入れず、経験則で 1 変数ずつ動かして実機で体感比較する。
- **対応**: `INTERP_DELAY` を 0.1s→0.15s に引き上げ（`game.ts`）。20Hz スナップ(50ms)に対し補間バッファの余裕を 1 間隔→2 間隔ぶんに拡大し、実回線ジッタによる `applyDirectSnapshot` フォールバック（カクつき/ワープ）を抑える。PvE 協力プレイのため +50ms の表示遅延は体感で無視でき、滑らかさを優先。バンド幅コストなし。
- **検証**: `tsc`/vitest 50件/`vite build` OK。本番 Pages 反映後にユーザーが実機で「結構ぬるぬる」＝滑らか・操作もっさり感なしを確認。
- **結論**: **3b は INTERP_DELAY=0.15 で確定**。次ダイヤル候補（未使用）＝カクつき残れば SNAP 20→30Hz／もっさりなら 0.12 へ。
- **対応**: Issue #88 → PR #89（main=`f7f3b01`）。relay 変更なし＝マージ（Pages）のみで本番反映。

## 2026-07-11 セッション: Co-op 味方タンク同士の当たり判定（押し出し）

- **範囲**: Co-op で味方戦車がすり抜けていた問題の解消。設計先行 → 実装 → ローカル検証 → CI → マージ → 本番反映。
- **最終 main**: `bcdc0eb`

### 味方タンク同士の当たり判定（§8 中8-b改定／中8-e新設）
- **背景**: プレイヤー戦車同士は当たり判定なし＝すり抜け。加えて敵の回避が `this.pos`＝ホスト自機のみ参照で、敵がゲスト戦車をすり抜けていた（§8 の穴。ソロは1人のため顕在化せず、Co-op で表面化）。
- **方針（ユーザー承認）**: 当初は「ブロック（壁と同じ）」で設計・実装したが、ユーザー要望で**押し出し方式**へ変更。
- **実装**:
  - `separatePlayers()` 新設。移動はブロックせず（味方に体当たりで進める）、毎フレーム移動後に重なった生存プレイヤー対を法線方向へ均等に押し離す。押し出し先が壁なら `slide`（壁ずり）でめり込ませない。`SEPARATE_ITERS=4` 反復で収束。死亡/待機中は対象外。
  - `movePlayer` のブロック条件から味方判定を除外（壁・敵は従来どおり）。
  - `moverBlocked`（敵の移動回避）を `this.pos` 単体→**全生存プレイヤー**ループに拡張＝敵のゲスト戦車すり抜けを解消。
  - 判定はホスト権威シムで実行＝Co-op で機能・ソロは1人のため無影響。スポーン間隔（最小64px > 判定48px）でハマらないことを確認。
- **検証**: `tsc`/vitest 50件/`vite build` OK。**ローカル（vite:5173 + wrangler dev:8787・ブラウザ2タブでホスト/ゲスト）で押し出し・壁際・敵回避をユーザー確認**（本番前検証）。
- **対応**: Issue #92 → PR #93（main=`bcdc0eb`）。relay 変更なし＝マージ（Pages）のみで本番反映。
